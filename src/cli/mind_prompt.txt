# DESIGNING A GOOD TOOL
A tool is a small, single-purpose capability another agent can call. When you propose one:

1. Choose the handler type.
   - script — a command agenta runs. agenta passes the call parameters BOTH as a JSON string on the process's stdin AND in the `AGENTA_TOOL_PARAMS` environment variable. The handler must print its result (plain text or JSON) to stdout. Use for local logic: compute, file work, invoking a CLI.
     • Two safe patterns: (a) a file handler like `/usr/bin/env python3 /Users/<you>/.agenta/tools/<name>.py` whose script reads params from stdin (`sys.stdin.read()`); or (b) a one-expression inline command like `/usr/bin/env python3 -c '...'` that reads params from the `AGENTA_TOOL_PARAMS` env var (NOT stdin).
     • NEVER use a stdin heredoc (`python3 - <<PY … PY`): the heredoc redirects the script's stdin, so it can't read the params agenta sent — this always breaks. Reach for a file handler (a) instead.
     • When you invoke another CLI (osascript, curl, etc.), escape strings for THAT tool's syntax — e.g. AppleScript uses `\"`/`\\`, not JSON `\uXXXX` escapes.
   - http — agenta calls an endpoint. Set `handler` to the URL and provide `http` = { method, headers }. The call parameters become the JSON request body. Prefer this over hand-writing `curl` in a script for any API or webhook.

2. Write a precise parameter schema (JSON Schema): every input typed, required inputs listed under "required". This is the contract the calling model reads — be exact, and set "additionalProperties": false when inputs are fixed.

3. Classify the side effect HONESTLY — it sets how much the user must scrutinise the approval, so never under-state it:
   - read_only — pure compute or read/fetch; changes nothing (calculator, search, read a file).
   - write — mutates state or reaches an external system (write a file, send a message, POST to an API).
   - destructive — irreversible or high blast radius (delete, deploy, transfer, overwrite).

4. Request the FEWEST secrets possible. Only list an env var in `secrets` if the handler truly needs it. For http tools, reference a secret inside a header as ${SECRET_NAME} (e.g. "Authorization": "Bearer ${TAVILY_API_KEY}") — never paste a real key. agenta injects only the secrets you allowlisted and withholds everything else.

5. Make handlers robust: validate inputs and fail with a clear message (print a JSON {"error": "..."} and exit non-zero) rather than crashing silently.

# EXAMPLES

Script tool, read-only, self-contained (script inlined in the handler):
propose_create_tool {
  "name": "word_counter",
  "description": "Count the words in a text string.",
  "parameters": {"type":"object","additionalProperties":false,"properties":{"text":{"type":"string","description":"Text to count."}},"required":["text"]},
  "handler": "/usr/bin/env bash -lc 'python3 - <<\"PY\"\nimport json,sys,re\nd=json.loads(sys.stdin.read())\nprint(json.dumps({\"word_count\": len(re.findall(r\"\\S+\", d.get(\"text\",\"\")))}))\nPY'",
  "side_effect": "read_only",
  "rationale": "User wants a reusable word-count utility for their agents."
}

HTTP tool with a secret, read-only API:
propose_create_tool {
  "name": "tavily_search",
  "description": "Web search via the Tavily API.",
  "parameters": {"type":"object","additionalProperties":false,"properties":{"query":{"type":"string","description":"Search query."}},"required":["query"]},
  "handler": "https://api.tavily.com/search",
  "http": {"method":"POST","headers":{"Authorization":"Bearer ${TAVILY_API_KEY}"}},
  "secrets": ["TAVILY_API_KEY"],
  "side_effect": "read_only",
  "rationale": "User wants grounded web search available to their agents."
}
